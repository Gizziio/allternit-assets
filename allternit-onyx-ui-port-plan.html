Priority Ports

Updated to reflect your feedback.
Port 01

Enhance the existing left shell rail

Do not remove most of the existing left-side shell rail. Keep it intact and add workflow helpers into or adjacent to it.

  • Add recents and pinned sessions.
  • Add pinned agents and project resume sections.
  • Add faster chat/project search and clearer active states.
Port 02

Packet-level streaming and inline streaming

Port the “live response anatomy” idea, not just token streaming. Show tool steps, search activity, reasoning, and answer flow inside the conversation.

  • Inline search/tool progress above the final answer.
  • Stable token streaming with fewer layout jumps.
  • Clear transition from work-in-progress to final response.
Port 03

Source and citation panel

Attach evidence to answers. Make provenance visible enough that users can audit trust without leaving the chat flow.

  • Cited sources ordered by appearance.
  • Found sources and uploaded files in one panel.
  • Answer-linked inspection, not separate admin browsing.
Port 04

Session architecture

Adopt a cleaner session model: sidebar session metadata, in-memory live session state, and optimistic creation for new sessions.

  • Separate session list from live message tree state.
  • Create sessions optimistically before first reply lands.
  • Auto-name sessions after first successful exchange.
Port 05

Tighter primary composer

Keep Allternit’s power, but reorganize it so the main composer feels intentional and operational.

  • Promote files, search, model, voice, and project context.
  • De-emphasize novelty or exploratory prompt actions.
  • Unify send behavior and active-state feedback.
Port 06

Conversation control polish

Make responses easier to retry, branch, regenerate, inspect, and recover after failure.

  • Edit and resend with less friction.
  • Support message branching and alternate answer paths.
  • Improve failure handling and “continue working” behavior.

Additional UI/UX Elements Worth Taking

Beyond the first recommendation set.
Extra

Command-style chat search

Onyx has a command menu approach for finding sessions and projects quickly.

  • Search chats and projects from one palette.
  • Use filters like “sessions” and “projects”.
  • Create a new session or project from the same interface.
Extra

Auto-naming and animated rename feedback

Onyx makes unnamed sessions feel temporary and then upgrades them as the conversation becomes real.

  • Show “unnamed” as a temporary state, not a dead end.
  • Auto-name after the first exchange.
  • Animate rename updates subtly in the session list.
Extra

Project-scoped chat grouping

Onyx keeps project chats grouped and sorted by latest activity, which is a strong pattern for Allternit.

  • Recent chats inside project context.
  • Move chats between projects cleanly.
  • Let projects act as a real work container, not just metadata.

Session Model To Take From Onyx

This is probably the biggest architecture lesson.

What Onyx appears to do

It treats sessions as three related but separate systems.

  • Server-backed session list for sidebar/history.
  • Local in-memory session store for live message trees and streaming state.
  • Optimistic pending sessions so a new chat exists immediately in the UI.

What Allternit should do

Adopt the same separation, translated into Allternit’s shell and thread model.

  • One durable session index for rail/sidebar/history.
  • One live session state layer for active thread rendering and packet flow.
  • One optimistic creation path so sessions appear before persistence finishes.

Behavior to port

  • Create a session before or at first send.
  • Navigate immediately to the new session route.
  • Auto-name after first meaningful reply.
  • Persist model override and session-level settings once the user actually uses them.
  • Keep per-session chat state: input, loading, streaming, uploading, tool-building.

Why it matters

  • Fixes “what is the current session?” ambiguity.
  • Makes sidebar state and live thread state agree more often.
  • Gives streaming a stable home instead of attaching it to transient UI state.
  • Makes branching, retry, and resume much easier to reason about.
